01

Decision Philosophy

Every design decision should reduce cognitive load, visual noise, engineering complexity, future maintenance, and design debt. Every design decision should increase trust, clarity, accessibility, predictability, reusability, and scalability.

Core Principles

01

User First

If it doesn't help the user, it doesn't belong

02

Problem First

Understand the problem before designing the solution

03

Clarity Before Beauty

Clear beats beautiful — always

04

Function Before Decoration

Every element must earn its place

05

Consistency Before Novelty

Predictability builds trust

06

Reuse Before Creation

Existing solutions are proven solutions

07

System Before Screen

Design the system, not just the screen

08

Long-term Thinking

Will this still work in 5 years?

Why These Principles Exist

Design debt accumulates silently. Every unnecessary component, every inconsistent pattern, every one-off token creates friction that compounds over time. These principles exist to prevent that accumulation. A design system is a living product. Like any product, it requires discipline to maintain.

User Value Framework

Before adding anything to the design system, ask these questions. If the answer to the first five is "No", do not add it.

The Value Test

QuestionIf YesIf No
Does this solve a real user problem?Consider addingDo not add
Will users notice its absence?Consider addingDo not add
Does it improve understanding?Consider addingDo not add
Does it reduce effort?Consider addingDo not add
Does it increase confidence?Consider addingDo not add

Examples

Good Decision — AI Insight Card
"Users need to understand why their spending increased. An AI Insight Card explains the reasoning with data. This solves a real problem."
Poor Decision — Decorative Illustration
"Users don't need an illustration to understand their balance. This adds visual noise without improving understanding."
03

Simplicity Framework

Always choose the simplest solution that satisfies user needs.

The Simplicity Test

QuestionIf YesIf No
Can this be removed?Remove itContinue
Can two components become one?Merge themContinue
Can two variants become one?Merge themContinue
Can one token replace three?Replace themContinue
Can Auto Layout solve this?Use Auto LayoutContinue
Can composition solve this?Use compositionContinue

Simplicity Hierarchy

1. Remove — Can it be eliminated entirely?
2. Merge — Can multiple things become one?
3. Reuse — Can an existing solution work?
4. Compose — Can existing pieces be combined?
5. Simplify — Can the existing solution be made simpler?
6. Create — Only as a last resort
04

Component Decision Tree

Only create a new component if every previous answer is "No". Every new component must have a documented reason.

Need a new component?
Does an existing component solve this?
YES → Use existing component
NO → Continue
Can an existing component be extended?
YES → Add variant/property to existing
NO → Continue
Can composition solve this?
YES → Compose existing components
NO → Continue
Can a variant solve this?
YES → Add variant to existing component
NO → Continue
Can a property solve this?
YES → Add property to existing component
NO → Continue
Create new component — Document reason for creation

NO → Continue

07

Visual Hierarchy Framework

Never create two primary actions with equal visual weight. Every screen must have exactly one primary insight and one primary action.

Hierarchy Levels

Primary Insight — The most important thing to know
Primary Action — The most important thing to do
Secondary Information — Important context
Supporting Information — Additional detail
Metadata — Reference information

Hierarchy Checklist

What is the primary insight?

What is the primary action?

Is the hierarchy immediately obvious?

Can anything be removed?

Can anything be simplified?

08

Motion Decision Tree

Every animation must communicate change. Never animate for decoration.

Need animation?
Does it communicate state?
YES → Consider animation
NO → Continue
Does it explain change?
YES → Consider animation
NO → Continue
Does it improve understanding?
YES → Consider animation
NO → Continue
Would removing it make experience worse?
YES → Keep animation
NO → Continue
Remove animation

Do

  • Communicate state change
  • Explain what happened
  • Keep animations fast (< 300ms)
  • Use subtle, purposeful motion

Don't

  • Animate for decoration
  • Animate to draw attention
  • Use slow, distracting animations
  • Add motion without purpose

AI Decision Framework

AI should never reduce transparency. AI must always explain its reasoning and be overrideable.

Need AI here?
Does AI genuinely help?
NO → Do not use AI
YES → Continue
Can rules solve this instead?
YES → Use rules
NO → Continue
Does the user understand why AI is involved?
NO → Explain AI involvement
YES → Continue
Can the recommendation be explained?
NO → Add explanation
YES → Continue
Can the user override it?
NO → Add override option
YES → Continue
Add AI with full transparency

Financial trust is more important than visual impact. Never exaggerate gains. Never hide risk.

YES → Continue

YES → Continue

NO → Add comparison
YES → Continue
Is uncertainty communicated?
NO → Add uncertainty indicator
YES → Continue

NO → Continue

Responsive Decision Framework

Never redesign simply because the screen is larger. Hierarchy must remain consistent across breakpoints.

Adapting layout?
Does hierarchy remain the same?
NO → Fix hierarchy first
YES → Continue
Does navigation remain familiar?
NO → Use standard navigation pattern
YES → Continue
Are important actions still visible?
NO → Prioritize actions
YES → Continue
Can interaction remain comfortable?
NO → Adapt interaction pattern
YES → Continue
Proceed with adaptation
13

Accessibility Decision Framework

Accessibility should influence every decision, not validate it afterwards. Every design must work without color.

Approving design?
Can VoiceOver complete this task?
NO → Fix VoiceOver support
YES → Continue
Can Dynamic Type display this correctly?
NO → Fix Dynamic Type support
YES → Continue
Can this be understood without color?
NO → Add non-color indicator
YES → Continue
Can keyboard users complete this flow?
NO → Fix keyboard support
YES → Continue
Approve design
